# 씨드핏 설계 검토 리포트 (Design Review)

**프로젝트**: 씨드핏 (Seed Fit) - 예산 맞춤 투자 분석 플랫폼  
**작성일**: 2026-04-26  
**검토 항목**: 로그인 설계 현황, 첫 화면 UX 전략, 모바일 퍼스트 전환 가능성

---

## 검토 항목 1: 로그인 기능의 TASK 기술 현황

### 1.1 현재 UI/UX의 로그인 부재

현재 프로토타입(`SeedFit_app/frontend`)에는 **로그인 화면이 구현되어 있지 않습니다.**  
이것은 버그가 아니라 **의도된 설계 결정**입니다.

### 1.2 TASK 문서에서의 명세 확인

`TASK` 상세명세서 및 `SRS-001 v1.2` 문서를 확인한 결과, 인증 관련 설계는 다음과 같이 정의되어 있습니다:

| 사용자 그룹 | 인증 방식 | TASK/SRS 근거 |
|---|---|---|
| **B2C (일반 사용자)** | **비로그인 전면 개방** | `C-TEC-009`: Supabase Auth 미사용. 임시 UUID 또는 브라우저 fingerprint로 사용자 식별 (MVP 단순화) |
| **B2B (공인중개사)** | **환경 변수 기반 시크릿 패스코드** | `SECRET_B2B` 환경 변수로 PasscodeGuard 컴포넌트를 통해 접근 제어 |
| **Admin (관리자)** | **환경 변수 기반 시크릿 패스코드** | `SECRET_ADMIN` 환경 변수로 PasscodeGuard 컴포넌트를 통해 접근 제어 |

**핵심 설계 원칙**: `C-TEC-009` 제약사항에 따라 **B2C는 비로그인 전면 개방**, B2B/Admin은 Supabase Auth가 아닌 **서버 환경 변수 기반 패스코드**로 단순화한 MVP 접근법을 채택하고 있습니다.

### 1.3 Phase 2에서의 인증 전환 계획

TASK 문서(`Batch07`, `Batch08`)에는 명시적으로 다음이 기술되어 있습니다:

> *"RISK-06 인지: 패스코드 방식은 MVP 임시 솔루션. Phase 2에서 Supabase Auth로 교체 예정"*  
> *"B2C 비로그인: 사용자 ID가 없으므로 임시 User 레코드 생성. Phase 2에서 인증 연동 시 머지 가능"*

### 1.4 결론

✅ **현재 로그인 기능 부재는 의도된 MVP 설계이며, TASK 문서에 명확히 기술되어 있습니다.**  
⚠️ 단, B2B 패스코드 게이트(`PasscodeGuard` 컴포넌트)는 아직 **구현이 완료되지 않은 상태**이므로 향후 개발 시 우선 적용이 필요합니다.

## 검토 항목 2: 첫 화면을 로그인으로 바꿀 것인가, 유지할 것인가?

### 2.1 현재 첫 화면의 설계 의도

SRS `REQ-FUNC-001`은 다음과 같이 명시합니다:

> *"앱 초기 랜딩 화면은 복잡한 지도가 아닌 **단일 '가용 현금 입력창'**을 화면 중앙에 배치해야 한다."*

이것은 제품의 핵심 차별점이자 **북극성 KPI(가용 현금 쿼리 검색 완료율 65%)를 달성하기 위한 핵심 퍼널 설계**입니다.

### 2.2 두 가지 시나리오 비교

#### ❌ 시나리오 A: 첫 화면을 로그인으로 전환

| 항목 | 평가 |
|---|---|
| 진입 장벽 | **매우 높음.** 회원가입·로그인 없이는 핵심 기능 체험 불가 → 신규 유저 이탈률 급증 |
| 퍼널 성과 | 북극성 KPI(`View_Landing_Page → Input_Cash_Amount → View_Filtered_Result`) 달성 **불가** |
| 브랜드 경험 | "3초 이내 역산"이라는 핵심 가치 제안을 첫 화면에서 전달하지 못함 |
| 로그인 데이터 활용 | B2C 사용자로부터 얻을 수 있는 데이터가 거의 없어 개인화 효용도 낮음 |
| TASK 일치 여부 | **현재 SRS/TASK 명세와 불일치** (대규모 요구사항 변경 필요) |

#### ✅ 시나리오 B: 현재 화면 유지 + 뒤에서 자연스럽게 로그인 유도 (권장)

| 항목 | 평가 |
|---|---|
| 진입 장벽 | **없음.** 즉시 핵심 기능 체험 가능 → 유저가 가치를 먼저 경험 |
| 퍼널 성과 | 북극성 KPI 달성에 최적화된 구조 |
| 브랜드 경험 | "예산만 입력하면 바로 결과"라는 강력한 첫인상 형성 |
| 로그인 유도 시점 | 매물 알림 구독(`Lead Gen`), Verified 매물 연락처 확인 등 **유저가 가치를 충분히 경험한 뒤 자연스럽게 유도** |
| TASK 일치 여부 | **현재 SRS/TASK 명세와 완전 일치** |

### 2.3 Best Practice 관점

글로벌 프롭테크 서비스 대다수(Zillow, Redfin, 직방 등)는 **"탐색 → 결과 확인 → 저장/알림/연락" 순서로 로그인 장벽을 최소화**하는 패턴을 채택합니다. 이는 프로덕트 Lead Gen 전략과도 정확히 일치합니다.

### 2.4 권장 로그인 유도 흐름

```
[랜딩: 가용 현금 입력] 
     ↓ 즉시 체험
[역산 결과 리스트 확인]
     ↓ 관심 구역 선택
[1:1 대조 대시보드]
     ↓ 매물 발견
[Verified 매물 연락처 클릭]
     ↓ 여기서 로그인/회원가입 유도 (Phase 2)
[알림 구독 / 중개소 연결]
```

### 2.5 결론

✅ **현재 화면(가용 현금 입력 랜딩)을 유지하고, 유저가 가치를 충분히 경험한 뒤 알림 구독·매물 연락 등의 시점에 로그인을 자연스럽게 유도하는 것이 압도적으로 권장됩니다.**

## 검토 항목 3: 모바일 퍼스트 전환 가능성 검토

### 3.1 현재 UI의 반응형 현황

현재 프로토타입은 **웹(데스크톱) 중심으로 구현**되어 있습니다.  
기술 스택(Next.js + Tailwind CSS)은 모바일 퍼스트 전환에 **기술적 장벽이 거의 없습니다.**

### 3.2 모바일 퍼스트가 적합한 이유

| 근거 | 내용 |
|---|---|
| **타겟 사용자** | 스마트 비기너(30~40대)는 부동산 정보 탐색을 **스마트폰으로 처음 접하는 비율이 높음** |
| **핵심 UX** | "가용 현금 입력 → 즉각 결과"는 모바일 환경에서도 단일 입력창으로 충분히 구현 가능 |
| **B2B 브리핑 모드** | SRS에 이미 **태블릿(B2B 브리핑 모드)** 지원을 명시하고 있어 모바일 반응형 설계가 전제됨 |
| **Lead Gen 전환** | 알림 구독·중개소 연락 등 전환 행동은 **모바일에서 더 자연스럽게 발생** |

### 3.3 전환 시 필요 작업 및 난이도

| 작업 항목 | 난이도 | 비고 |
|---|---|---|
| Tailwind 반응형 유틸(`sm:`, `md:`, `lg:`) 적용 | 🟢 낮음 | 클래스 수정으로 대부분 해결 가능 |
| 1:1 대조 대시보드 레이아웃 재설계 | 🟡 중간 | 좌우 분할 → 상하 스크롤 구조로 전환 필요 |
| `DeepAnalysisReport` 테이블 모바일 최적화 | 🟡 중간 | 가로 스크롤 또는 카드형 전환 필요 |
| B2B 브리핑 모드 태블릿 레이아웃 | 🟡 중간 | 태블릿(768px~) 기준으로 별도 뷰포트 설계 필요 |
| 터치 인터랙션 최적화 (버튼 크기, 탭 타겟) | 🟢 낮음 | Tailwind `min-h-[44px]` 등 표준 적용 |
| 성능 최적화 (이미지, 번들 사이즈) | 🟡 중간 | Next.js Image 컴포넌트 활용으로 대부분 대응 가능 |

### 3.4 전환 전략 제안

**점진적 모바일 퍼스트 전환**을 권장합니다:

1. **Phase 1 (즉시 적용 가능)**  
   - 랜딩 페이지(가용 현금 입력창) → 모바일 최적화 우선 적용  
   - 버튼·입력창 터치 타겟 크기 보정  
   - Tailwind 반응형 Breakpoint 전면 적용

2. **Phase 1 후반 (핵심 화면)**  
   - 1:1 대조 대시보드 → 모바일 세로형 카드 레이아웃으로 재설계  
   - `DeepAnalysisReport` → 탭 방식으로 섹션 분리

3. **Phase 2 (고도화)**  
   - PWA(Progressive Web App) 전환으로 홈 화면 추가, 알림 푸시 지원  
   - 모바일 전용 알림 구독 UX 강화

### 3.5 결론

✅ **모바일 퍼스트 전환은 적합하며, 기술적 어려움은 낮습니다.**  
현재 Tailwind CSS + Next.js 스택은 반응형 구현에 최적화되어 있어 **전면 재구현 없이 점진적 전환이 가능합니다.**  
단, 1:1 대조 대시보드와 Deep Analysis Report는 레이아웃 재설계가 필요하며, 이를 Phase 1 후반 작업으로 계획에 반영하는 것을 권장합니다.
